## New Findings

### 1. Punctuation Silently Kills Augmentation Coverage

**File:** `backend/data_pipeline/augment.py`, line 139
**Severity:** High

`augment_by_substitution` checks `ko.endswith('아요'/'어요'/'여요')` to find the terminal
morpheme. If the sentence ends with punctuation (e.g. `'드세요?'`), all three checks fail
and neither a formal nor casual variant is generated for that pair. No warning is logged.

Subtitle data commonly terminates sentences with `?` or `.`. A large share of polite
pairs silently contributes nothing to augmentation, skewing the formal and casual class
counts lower than expected.

**Fix:** Strip trailing punctuation before the endswith check, or strip it from `ko`
before the suffix scan and re-attach it to the substituted output.

```python
import re
_TRAILING_PUNCT = re.compile(r'[.?!]+$')

def augment_by_substitution(pairs):
    for en, ko, formality in pairs:
        if formality != 'polite':
            continue
        ko_stripped, punct = _TRAILING_PUNCT.subn('', ko)  # remove trailing punct
        # ... suffix matching on ko_stripped ...
        # re-attach punct to the output
```

### 2. -지요 Endings Systematically Mislabeled as Casual

**File:** `backend/data_pipeline/label.py`, line 37
**Severity:** High

When Mecab parses a `-지요` sentence-final, it tags `지` as EF and `요` as JX (particle).
`extract_ef_morpheme` returns `'지'`, and `label_formality('지')` matches the casual
rule `'-지'`. The polite `-요` fallback at line 64 is only reached when **no EF morpheme
is found at all** — it never fires here because `지` was already tagged as EF.

`-지요` is a polite/softening ending (해요체 equivalent). Sentences like
`'알겠지요'` ("I understand") are labeled casual and routed to the wrong training class.

**Fix:** Add `-지요` as an explicit polite suffix rule checked before the bare `-지` rule,
or handle it as a compound morpheme in the EF extraction step.

```python
# In SUFFIX_RULES, before the casual '-지' entry:
("-지요", "polite"),
("-죠",   "polite"),
```

### 3. '여요' → '여' Produces Ungrammatical Korean

**File:** `backend/data_pipeline/augment.py`, line 35
**Severity:** High

`POLITE_TO_CASUAL` maps `'여요'` → `'여'`. The ending `'여'` is not a valid Korean
casual sentence-final morpheme — it is a connective or exclamatory particle, not a
terminating form. `augment_by_substitution` performs no morphological verification
(unlike `augment_by_llm`), so these ungrammatical pairs enter `train.tsv` directly.

**Fix:** Either remove the `'여요'` mapping entirely (it is rare in subtitle data and
covers only a handful of verbs like `이다` → `이에요/이여요`), or map it to `'야'`
which is the correct casual copula ending.

```python
POLITE_TO_CASUAL = {
    "아요": "아",
    "어요": "어",
    # "여요": "여",  # '여' is not a valid casual ending — skip or map to '야'
}
```

### 4. StopIteration Crashes Translate and Set-Context on API Refusals

**File:** `backend/api/agents/claude_agent.py`, lines 86 and 207
**Severity:** High

Both `parse_situation()` and `translate()` extract the tool-use block with:
```python
next(b for b in response.content if b.type == "tool_use").input
```
There is no default argument to `next()`. If the API returns a safety refusal or
truncates the response at `max_tokens` before emitting the tool-use block, the generator
is exhausted and `StopIteration` propagates uncaught. FastAPI wraps it as HTTP 500 with
`detail='Translation failed: '` — the empty detail string hides the root cause.

**Fix:**
```python
tool_block = next(
    (b for b in response.content if b.type == "tool_use"), None
)
if tool_block is None:
    raise ValueError(f"No tool_use block in response (stop_reason={response.stop_reason})")
result = tool_block.input
```

### 5. LLM-Generated Single Pairs Bypass Length Filtering

**File:** `backend/data_pipeline/augment.py`, `_parse_and_verify` function (~line 595)
**Severity:** Medium-High

`_parse_and_verify_triplets` explicitly calls `is_valid_pair` on all three Korean
sentences before accepting a triplet. `_parse_and_verify` (used by `augment_by_llm`
for single-register pairs) only checks `label_sentence` and `_MIN_EN_WORDS` — it never
calls `is_valid_pair`. LLM-generated pairs with over-long Korean sentences (e.g.
42 Mecab morphemes, above the formal class cap of 30) enter `train.tsv` directly.

**Fix:** Add an `is_valid_pair` call inside `_parse_and_verify`:

```python
from .filter import is_valid_pair

# inside _parse_and_verify, after the label check:
if label == expected_label and is_valid_pair(en, ko, label):
    verified.append((en, ko, label))
```

### 6. Inference Doesn't Pin tokenizer.src_lang

**File:** `backend/model/inference.py`, line 41
**Severity:** Medium

`MBartInference.translate` calls the tokenizer without first setting
`self.tokenizer.src_lang = 'en_XX'`. mBART-50 prepends the `src_lang` token to every
encoded sequence. If any caller mutates `tokenizer.src_lang` before calling `translate`
(e.g. a debug eval loop, a parallel dataset construction), the English source is encoded
with the wrong language prefix and the model produces degraded or wrong-language output
with no error.

**Fix:**
```python
self.tokenizer.src_lang = "en_XX"
inputs = self.tokenizer(conditioned_input, return_tensors="pt", ...)
```

### 7. Formality Accuracy Denominator Excludes Unclassifiable Predictions

**File:** `backend/evaluation/metrics.py`, line 54
**Severity:** Medium

`formality_accuracy` computes `correct / classifiable` (where `classifiable = total - none_count`)
rather than `correct / total`. Predictions where Mecab finds no sentence-final morpheme
are dropped from both numerator and denominator. This inflates FA when the model
degenerates to producing noun fragments or short non-verbal outputs.

A model generating unclassifiable output for 30% of examples and correct formality for
85% of the remaining 70% reports FA=85% but satisfies the formality constraint on only
~60% of all inputs. The published training result of FA=85.12% may overstate compliance.

**Fix:** Track two metrics — `fa_classifiable` (current behavior, useful for diagnosis)
and `fa_total` (`correct / total`, the honest headline number).

### 8. Hypothesis/Reference Misalignment in evaluate.py

**File:** `backend/model/evaluate.py`, line 75
**Severity:** Medium

`evaluate_model` generates hypotheses by iterating `load_split(test_tsv, tokenizer)`
and loads references via a separate `_load_rows(test_tsv)` call. The two readers apply
different filtering logic. The only guard is a row-count equality check (`len(hypotheses)
!= len(references)`). If the two readers filter the same total number of rows but
different specific rows, the length check passes and `hypotheses[i]` is silently matched
against the wrong `references[i]`.

**Fix:** Generate hypotheses and load references in a single pass over the same rows, or
load references from the same `load_split` call that produces the generation inputs.

### 9. PUBLIC Setting Over-Formalizes Any Relationship

**File:** `backend/formality/resolver.py`, line 79
**Severity:** Medium

```python
if rel == RelationshipType.STRANGER or setting == SettingType.PUBLIC:
    return FormalityToken.FORMAL
```

This rule fires **before** the casual rules, so a friend or colleague encountered in
a public setting always resolves to `<formal>` regardless of relationship. By Korean
pragmatic norms, running into a work colleague at a café (relationship=colleague,
setting=public) warrants polite (-아요/어요), not formal (-습니다).

**Fix:** Narrow the public rule to strangers and explicitly unknown relationships only,
or demote it to a polite default rather than formal:

```python
if rel == RelationshipType.STRANGER:
    return FormalityToken.FORMAL
if setting == SettingType.PUBLIC and rel in (RelationshipType.ACQUAINTANCE,):
    return FormalityToken.POLITE  # not FORMAL
```

### 10. forced_bos_token_id Conflict Crashes Evaluation on transformers ≥ 4.46

**File:** `backend/model/evaluate.py`, line 90
**Severity:** Medium

`evaluate_model` passes `forced_bos_token_id=ko_id` explicitly to `model.generate()`
while `model.generation_config.forced_bos_token_id` is also set from training. In
`transformers >= 4.46`, passing a generate-call argument that conflicts with
`generation_config` raises:
```
ValueError: You are explicitly passing `forced_bos_token_id` to generate but it was
already set in generation_config.
```
This crash happens **after** the full translation loop completes (60+ minutes on a T4),
losing all results. `inference.py` already handles this correctly by not re-passing
the argument.

**Fix:** Remove the explicit `forced_bos_token_id` argument from the `generate` call in
`evaluate.py` and rely on `generation_config` as `inference.py` does.
